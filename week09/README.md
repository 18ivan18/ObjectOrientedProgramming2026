## Теория: `static` и templates

### `static` (статични член-данни и методи)

`static` член-данните са “общи” за всички обекти от даден клас — има **точно една** стойност, споделена от всички инстанции. Полезни са за броячи, кеширане, константи на класа и общо състояние.

`static` методите са функции, които **принадлежат на класа**, а не на конкретна инстанция. Те нямат `this` и могат да достъпват директно само `static` членове. Често се използват като _factory methods_ (например `Error::newNone()`), помощни функции, или за създаване на обекти в валидно състояние.

Пример (брояч на живи обекти):

```c++
#include <iostream>

class Student {
    static int aliveCount; // декларация (една стойност за всички Student)
public:
    Student() { ++aliveCount; }
    Student(const Student&) { ++aliveCount; }
    ~Student() { --aliveCount; }

    static int getAliveCount() { return aliveCount; } // static метод
};

int Student::aliveCount = 0; // дефиниция (в .cpp файл)

int main() {
    Student a;
    Student b = a;
    std::cout << Student::getAliveCount() << "\n"; // 2
}
```

Също ще срещате `static` локални променливи във функции: те се инициализират **само веднъж** и пазят стойност между извикванията.

```c++
int nextId() {
    static int id = 0;
    return ++id;
}
```

### Templates (шаблони / generics)

Template-ите позволяват да пишем “код по шаблон”, който работи за много типове, без да дублираме логика. Компилаторът генерира конкретна версия, когато template-ът се използва с даден тип (например `Matrix<int>`, `Matrix<double>`).

Има два най-чести вида:

- **Function templates** — шаблонни функции (например “максимум”, “сравнение”, “сума”).
- **Class templates** — шаблонни класове (например `Vector<T>`, `Optional<T>`, `Matrix<T>`).

Пример за шаблонна функция:

```c++
template <typename T>
T maxOf(const T& a, const T& b) {
    return (a < b) ? b : a;
}

int main() {
    int x = maxOf(3, 7);            // T = int
    double y = maxOf(2.5, 1.1);     // T = double
}
```

Пример за шаблонен клас:

```c++
template <typename T>
class Box {
    T value;
public:
    Box(const T& v) : value(v) {}
    const T& get() const { return value; }
};

int main() {
    Box<int> a(5);
    Box<const char*> b("hello");
}
```

Важно: понеже компилаторът трябва да “вижда” тялото на template-ите при инстанциране, шаблонните класове/функции обикновено се реализират **в header** (или в `.hpp` включван от header).

## Задача 1

Напишете клас `UniqueInstance`, който дава на всяка своя инстанция **уникално `id`**, което се увеличава с 1 за всяка новосъздадена инстанция.

Изисквания:

- Класът да има `static` член-данна (например `nextId`), която пази следващото свободно `id`.
- Всеки обект да пази своето `id` (което не се променя след създаване).
- При създаване на обект (включително и при копиране) той трябва да получава **ново** `id`.
- Да има метод `getId()`, който връща `id`-то на обекта.
- Да има `static` метод `getNextId()`, който връща стойността на `nextId` (за дебъг/тестване).
- Имплементирайте **голямата четворка** така, че правилата по-горе да се спазват:
  - default constructor
  - copy constructor
  - operator= (не трябва да променя `id`, защото обектът вече съществува)
  - destructor

Примерен тест (не е задължително да изглежда точно така):

```c++
int main() {
    UniqueInstance a;
    UniqueInstance b;
    UniqueInstance c = a; // нов обект => ново id
    b = a;                // няма нов обект => id на b не се променя

    std::cout << a.getId() << " " << b.getId() << " " << c.getId() << "\n";
}
```

# Задача 5

Имплементирайте опростени версии на умните указатели:

- `MyUniquePtr<T>` (уникална собственост)
- `MySharedPtr<T>` (споделена собственост чрез reference counting)
- `MyWeakPtr<T>` (слаба референция към `MySharedPtr`)

Минимални изисквания:

### `MyUniquePtr<T>`

- Да пази указател `T*`.
- Да **не може** да се копира (изтрийте copy ctor и copy `operator=`).
- Да може да се мести (move ctor и move `operator=`).
- В деструктора да освобождава ресурса с `delete`.
- Да има: `get()`, `operator*`, `operator->`, `release()`, `reset(T* = nullptr)`.

### `MySharedPtr<T>`

- Да пази:
  - указател към обекта `T*`
  - указател към контролен блок / броячи (поне `sharedCount`, по желание и `weakCount`)
- Copy конструктор/оператор= да увеличават `sharedCount`.
- Деструкторът да намалява `sharedCount` и:
  - при `sharedCount == 0` да изтрие управлявания обект (`delete T*`)
  - при `sharedCount == 0` и `weakCount == 0` (ако имате weak) да изтрие и контролния блок
- Да има: `use_count()`, `get()`, `operator*`, `operator->`, `reset()`, `swap(...)`.

### `MyWeakPtr<T>`

- Да може да се създава от `MySharedPtr<T>` (увеличава `weakCount`).
- Да **не държи обекта жив** (т.е. не увеличава `sharedCount`).
- Да има:
  - `expired()` (истина ако `use_count() == 0`)
  - `lock()` (връща `MySharedPtr<T>` ако обектът съществува, иначе празен `MySharedPtr<T>`)

Примерен `main()`:

```c++
#include <iostream>
// #include "MyUniquePtr.h"
// #include "MySharedPtr.h"
// #include "MyWeakPtr.h"

struct Foo {
    int x;
    Foo(int x) : x(x) {}
    ~Foo() { std::cout << "~Foo\n"; }
};

int main() {
    {
        MyUniquePtr<Foo> u(new Foo(10));
        std::cout << u->x << "\n";
        MyUniquePtr<Foo> u2 = std::move(u);
        std::cout << (u.get() == nullptr) << "\n";
    } // deletes Foo

    MySharedPtr<Foo> s1(new Foo(20));
    std::cout << s1.use_count() << "\n"; // 1
    {
        MySharedPtr<Foo> s2 = s1;
        std::cout << s1.use_count() << "\n"; // 2

        MyWeakPtr<Foo> w = s1;
        std::cout << w.expired() << "\n"; // 0 (false)
        MySharedPtr<Foo> s3 = w.lock();
        std::cout << s3.use_count() << "\n"; // 3 (примерно)
    }
    std::cout << s1.use_count() << "\n"; // 1
    s1.reset(); // deletes Foo (ако няма други shared)
}
```

## Задача 6

Асистентите по ООП са решили да тестват автоматично задачите на студентите тази година и се нуждаят от вашата помощ да напишат системата си.

Класовете, които трябва да имплементирате: `Error`, `TestCase`, `TestSuite`.

Всяка задача по ООП ще си има собствен набор от тестове наричан `TestSuite` (Тестов пакет). Той трябва да има:

- Име (може да се променя)
- Множество тестови сценарии (използвайте написания от вас клас `Vector<T>` / `Vector` от седмица 4)
- Възможност за изпълнение на всички тестови сценарии
- Възможност за добавяне на нови сценарии към множеството
- Възможност за филтриране на преминаващи/непреминаващи сценарии
- Възможност за филтриране на сценариите по тип грешка
- Възможност за премахване на всички тестови сценарии с даден тип грешка
- Възможност за цветно принтиране на информация

`TestCase` (тестов сценарий) представлява отделен тест за точно една функционалност. Всеки сценарий трябва да има:

- Име
- Набор от функции за тестване (засега ще напишем само 3 лесни такива):
  - `assertTrue` — приема булев аргумент и проверява дали стойността му е истина
  - `assertEquals` — приема два аргумента и проверява дали са равни по стойност
  - `assertNotNull` — приема указател към обект и проверява, че той не е `nullptr`
- Възможност да се провери дали теста е бил успешен
- Възможност да се провери дали има грешка
- Възможност да се провери типа на грешката
- Възможност да се провери съобщението на грешката

Всеки тест може да е успешен, а може и да не е. Когато един тест не е минал успешно е необходимо да имаме допълнителна информация какво се е счупило. За тази цел тестовите сценарии могат да имат грешка, която трябва да има:

- Тип (Празна/Никаква грешка, Грешка при компилация, Предупреждение, Неуспех при сравнение)
- Съобщение
- Възможност да се проверява типа на грешката
- Възможност да се проверява дали грешката има съобщение
- Възможност да се проверява съобщението на грешката ако има такова
- Възможност да се създава нова грешка от всеки един тип

Пример за factory methods в `Error`:

```c++
static Error newNone();
static Error newBuildFailed(const char*);
static Error newWarning(const char*);
static Error newFailedAssertion(const char*);
```

## Задача 7

Съществуват няколко начина за обработване на грешки в различните езици за програмиране. Тук ще разгледаме един от тях.

## Error

Даден е клас `Error`, който пази в себе си съобщение за грешка.

## Optional

Даден е клас `Optional<T>`, който може да съдържа даден елемент, но може и да е празен.

Проверката дали даден `Optional` обект е празен, се извършва от метода `isNone()`

Можем да вземем стойността на `Optional` обект (дори и да е празен), чрез метода `getValue()` - ако обекта е празен, пак се очаква да върнем обект празен от тип `T` .

Един `Optional<T>` е празен, когато не му подадем аргумент в конструктора.

## Result

Даден е клас `Result<T>`, който може да съдържа даден резултат, или пък грешка (от тип `Error`).

Предефинирайте оператора за сравнение (==), така че да приема `Error` или `T` и да връща дали дадения `Result` обект е грешка или не.

Напишете методи, за връщане на резултата и грешката, като използвате `Optional` (все пак, може и да нямаме резултат, или пък да нямаме грешка) .

## Пример за оператор== при Result:

Result<int>(3) == int() -> true
Result<int>(5) == Error() -> false
Result<int>(“Error Message”) == Error() -> true
